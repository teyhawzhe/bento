## Context

A001 目前已提供員工與管理員登入、忘記密碼、修改自己的密碼，以及管理員新增、匯入、停用與重設員工帳號的能力。最新的 `A001.md` 進一步要求管理員可以直接修改既有員工資料，包含 `username`、`name`、`email`、`department_id` 與 `is_admin`，且 `id` 必須維持唯讀。這代表 A001 的員工管理能力從「建立後只能停用或重設」擴充為「可完整維護員工主檔」。

目前系統中，員工資料已與 department 主檔關聯，且後台已存在員工清單與部門資料來源，因此這次設計重點是補上更新契約、唯一性驗證、權限限制與前端編輯互動，而不是重新定義員工或部門資料模型。

## Goals / Non-Goals

**Goals:**
- 提供管理員修改既有員工資料的 API 與畫面操作流程。
- 讓管理員可編輯 `username`、`name`、`email`、`departmentId` 與 `isAdmin`。
- 保持 `id` 為唯讀欄位，不允許更新。
- 更新時驗證 `username`、`email` 與其他員工不可重複，並驗證 `departmentId` 必須指向有效部門。
- 保持 A001 既有登入、停用、重設密碼流程不被破壞。

**Non-Goals:**
- 不調整員工登入、管理員登入、忘記密碼或修改自己密碼的契約。
- 不新增員工刪除功能。
- 不重新設計 A010 部門管理流程，只重用既有部門主檔能力。
- 不變更員工資料表主鍵或 JWT 結構。

## Decisions

### 1. 使用 `PATCH /api/admin/employees/{id}` 作為更新入口
- 理由：`A001.md` 已明確指定此 API，且更新既有員工資料屬於部分欄位更新語意，使用 `PATCH` 最符合目前 A001 風格。
- 替代方案：改用 `PUT /api/admin/employees/{id}`。未採用，因為目前需求並未要求整筆覆蓋，也會讓前端必須送出更多固定欄位。

### 2. 維持 `id` 唯讀，只允許修改指定欄位
- 理由：`id` 是員工主鍵，應視為識別值而非業務可編輯欄位；`departmentId`、`isAdmin` 則屬於可被管理員維護的主檔資訊。
- 替代方案：允許任意欄位更新。未採用，因為會讓資料契約過於寬鬆，增加風險。

### 3. 唯一性驗證採「排除自己」邏輯
- 理由：更新員工時，若 `username` 或 `email` 未變更，不應被自己的既有資料判定為重複；因此 repository 或 service 必須在驗證時排除同一筆員工。
- 替代方案：沿用新增員工的純 exists 驗證。未採用，因為會造成任何更新都可能被自己擋下。

### 4. 前端在員工清單內提供編輯入口
- 理由：目前後台已有員工清單與各員工卡片，直接在同一區塊展開或編輯既有資料，操作路徑最短，也符合 A001 作為管理工具頁的使用情境。
- 替代方案：另開獨立頁面或 modal。未採用，因為這次需求範圍不大，留在既有區塊可降低版面與路由複雜度。

## Risks / Trade-offs

- [更新邏輯與新增邏輯的唯一性規則不同] → 在 service 層明確拆分 create 與 update 驗證，避免共用錯誤的 exists 判斷。
- [管理員可將一般員工改為管理員，可能影響權限] → 明確將 `isAdmin` 納入受控欄位，僅允許已驗證管理員操作，並在後端強制驗證 JWT role。
- [前端若直接編輯列表資料，容易造成暫存狀態與實際資料不同步] → 使用獨立編輯 state，送出成功後重新載入員工清單。
- [既有 UI 只支援停用與重設密碼，加入編輯後畫面複雜度上升] → 將編輯欄位集中在同一卡片或清楚區塊中，避免與狀態切換混在一起。

## Migration Plan

- 先更新 OpenSpec 與測試，再補後端 API、service、repository。
- 前端在既有員工管理畫面中加入編輯欄位與儲存動作。
- 驗證 `PATCH /api/admin/employees/{id}`、既有新增/匯入/停用/重設密碼功能都正常。
- 若需 rollback，可先移除前端編輯入口並停用新 API，不影響既有登入與員工建立流程。

## Open Questions

- `department` 更新後，員工清單是否要立即顯示新的 `department.name`：目前假設送出成功後重新載入清單即可。
- 是否允許管理員將自己從 `isAdmin=true` 改為 `false`：`A001.md` 目前沒有額外限制，先依需求允許，但實作時可再決定是否加保護。
