## Context

最新 A006 已將「查詢供應商清單及其對應的便當選項」提升為管理員查詢能力的一部分，並明確列出 `GET /api/admin/suppliers`。目前系統的供應商管理與菜單管理雖然已在 frontend 加強關聯顯示，但實際上仍依賴 `/suppliers` 與 `/menus` 兩支既有 API 自行拼接，尚未符合新的 A006 契約。

因此這次 change 不只是 UI 微調，而是要把供應商與其便當選項的查詢能力正式納入管理員流程：由 backend 提供聚合後的供應商查詢結果，再讓 frontend 以該資料驅動供應商視角與菜單辨識體驗。

## Goals / Non-Goals

**Goals:**
- 讓管理員可透過 A006 定義的查詢能力取得「供應商 + 對應便當選項」資料。
- 讓管理員在選取供應商時，可以直接看到該供應商底下的菜單清單。
- 讓菜單清單中的每筆菜單更容易辨識其所屬供應商，不再只依賴 `supplierId`。
- 讓建立菜單流程可直接沿用供應商上下文，減少手動輸入供應商 ID。
- 讓供應商管理畫面以 `供應商` / `菜單` 分頁區分不同工作內容，降低單一畫面過於混雜的負擔。

**Non-Goals:**
- 不修改供應商或菜單的資料表結構。
- 不重做整個管理員資訊架構，只調整供應商管理區塊內的資訊架構與供應商/菜單關聯呈現。
- 不變更供應商或菜單的既有 CRUD 權限與驗證規則。

## Decisions

### 1. 新增管理員供應商聚合查詢 API
- 決策：新增或調整 `GET /api/admin/suppliers`，讓 backend 直接回傳供應商清單，每筆資料都包含對應的便當選項。
- 原因：這與最新 A006 的 API 契約一致，也能避免前端長期自行拼接兩份資料。
- 替代方案：維持 frontend 以 `/suppliers` + `/menus` 拼接。未採用，因為這已和最新 A006 脫節。

### 2. 供應商詳情區直接使用聚合結果顯示便當選項
- 決策：供應商詳情區不再自行推算關聯菜單，而是直接讀取該供應商在管理員供應商查詢結果中的便當選項。
- 原因：這可讓 UI 與 A006 的「供應商清單及其便當選項」保持一致。
- 替代方案：僅在前端計算關聯。未採用，因為會讓規格、API 與 UI 三者再度分離。

### 3. 菜單清單主動顯示供應商名稱
- 決策：菜單清單每筆資料要把供應商名稱作為主要資訊的一部分顯示，必要時保留 `supplierId` 作為次要輔助。
- 原因：名稱比數字 ID 更符合管理員的日常辨識方式。
- 替代方案：維持只顯示 `supplierId`。未採用，因為這正是目前辨識成本高的來源。

### 4. 建立菜單流程改為選擇供應商，而不是手填 ID
- 決策：建立菜單區改用供應商下拉選單，並在已選供應商時可預設帶入該供應商。
- 原因：這能減少輸入錯誤，也讓「先選供應商，再管理其便當」的工作流更順。
- 替代方案：保留文字輸入 `supplierId`。未採用，因為這要求管理員先記住 ID，再手動配對。

### 5. 供應商管理區內改用 `供應商` / `菜單` 工作分頁
- 決策：在 `admin-suppliers` 主頁籤內再拆成兩個次分頁，`供應商` 分頁負責查詢、新增、更新供應商與查看供應商關聯菜單；`菜單` 分頁負責建立、查詢與編輯菜單。
- 原因：目前把供應商主檔與菜單維護都放在同一個工作區會讓畫面焦點分散，分頁能讓管理員知道自己正在維護哪一層資料。
- 補充：分頁上方固定顯示目前選取的供應商脈絡；若尚未選取供應商，菜單分頁仍可從下拉選單選擇，但需明確提示目前未鎖定任何供應商。
- 替代方案：只保留單頁式布局。未採用，因為這正是使用者覺得「畫面不直覺」的來源。

## Risks / Trade-offs

- [新增聚合 API 後，frontend 既有供應商管理資料流需要重接] → 用新的 admin suppliers DTO 隔離影響面，逐步替換現有供應商視圖資料來源。
- [在同一頁放入更多關聯資訊，可能讓供應商詳情區過於擁擠] → 優先顯示精簡卡片與清楚標題，避免再塞入過多欄位。
- [建立菜單改成下拉選單後，既有習慣手打 ID 的使用者需要適應] → 保留供應商編號作為輔助標示，降低切換成本。
- [新增次分頁後，使用者可能在切換時失去供應商上下文] → 在分頁頂部固定顯示目前選取供應商，並在菜單分頁內保留當前帶入提示。

## Migration Plan

- 先更新 `admin-order-management`、`bento-supplier-management` 與 `bento-menu-management` 的 delta specs，固定 A006 的新查詢能力與 UI 行為。
- 實作時先補 backend `GET /api/admin/suppliers` 聚合查詢，再調整 frontend state、分頁與畫面。
- 若需回滾，可恢復文字輸入 `supplierId` 與原有列表呈現；backend 無資料 migration 需求。

## Open Questions

- `GET /api/admin/suppliers` 是否需要支援名稱搜尋或只回傳全部清單；A006 目前僅要求查詢供應商清單及其便當選項。
