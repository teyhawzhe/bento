## Context

A009 來自 `../uml/input/A009.md`，要求前端在登入後依角色直接導向對應 TAB 主頁，並將成功、失敗、警告、確認四類提示統一成 MessageBox 彈窗。目前 `frontend/src/App.tsx` 仍以單一大型元件承載登入、員工功能與管理員功能，且在先前版本曾保留「員工入口 / 管理員入口」切換概念；這與最新版 UML 中「不顯示入口選擇頁」的要求不一致。

這種結構雖然已能支援 A002、A004、A006、A007、A008 的流程，但會讓登入後介面切換成本高，也讓各操作的提示風格不一致。A009 的核心不是改動後端資料或 API，而是建立一個前端共用互動殼層，讓既有能力能在一致的導覽與回饋模式下運作。

## Goals / Non-Goals

**Goals:**
- 建立登入後依角色顯示的 TAB 導覽列，切換功能內容時不重新整理整頁。
- 移除登入前的角色入口切換，改為使用單一登入表單並在登入成功後直接依角色導向對應 TAB 主頁。
- 將員工頁整理為「訂便當」與「修改密碼」兩個主頁籤，並把「我的訂單」併入「訂便當」TAB 右側同步顯示。
- 將管理員頁整理為「訂單管理」、「供應商管理」、「報表設定」與「系統設定」四個主頁籤，並把既有功能對應到正確區塊。
- 導入共用 MessageBox 狀態與元件，統一處理成功、錯誤、警告與刪除確認。
- 保持既有 API 契約與大部分業務邏輯不變，優先重構畫面結構與互動回饋方式。

**Non-Goals:**
- 不在 A009 內新增或修改後端 API。
- 不重做整體視覺主題或品牌設計；登入區僅調整成單一入口，不擴充新的帳號流程。
- 不把 `App.tsx` 一次完整拆成大量功能檔案；若需要抽離，優先抽共用 TAB 與 MessageBox 元件以及少量區塊元件。
- 不變更 A002、A004、A006、A007、A008 的業務規則，例如截止時間、驗證規則或查詢條件。

## Decisions

### 1. 以單一前端 shell state 管理登入後 TAB
- 決策：在 `App.tsx` 或其拆出的 shell 元件內新增 `activeTab` 狀態，依 `session.role` 初始化對應預設頁籤，點擊 TAB 時僅切換前端顯示內容。
- 原因：A009 明確要求「點擊 TAB 切換功能頁面，不重新整理整頁」，因此以 client-side state 管理最直接，也不需要新增 router 相依或調整部署設定。
- 替代方案：引入 React Router 以路由切頁。未採用，因為本次需求只需要登入後固定頁籤切換，加入路由會增加結構改動與測試面積。

### 1-1. 登入流程改為單一表單，自動判斷角色入口
- 決策：前端保留單一登入表單，不再讓使用者先選「員工入口 / 管理員入口」；送出時先嘗試員工登入入口，若後端明確回覆「請使用管理員登入入口」，再自動改呼叫管理員登入 API。
- 原因：最新版 A009 要求登入成功後依 JWT 角色直接進入對應主頁，且不可再顯示入口選擇頁；在不變更既有後端雙登入 API 的前提下，前端自動判斷是最低風險方案。
- 替代方案：後端改成單一 `/auth/login` 再由服務端判斷角色。未採用，因為 A009 聚焦前端介面規範，先不擴張到 A001 API 契約調整。

### 2. 以角色映射定義固定頁籤與對應內容區塊
- 決策：建立員工與管理員各自的 tab definition，員工固定顯示「訂便當」「修改密碼」，管理員固定顯示「訂單管理」「供應商管理」「報表設定」「系統設定」。
- 原因：這樣可以把 A009 的頁籤規格集中管理，也能避免以條件散落在 JSX 中導致後續維護困難。
- 替代方案：直接在 JSX 手寫多段按鈕與條件渲染。未採用，因為目前 `App.tsx` 已相當長，應避免再增加分散邏輯。

### 2-1. 員工端將下單與個人訂單整合在同一工作面
- 決策：把「我的訂單」從獨立 TAB 改成「訂便當」TAB 右側的同步清單，左側維持下單流程，右側保留取消訂單操作。
- 原因：這樣更符合員工操作心智模型，送出訂單後能立刻在同一視野確認結果，也不用來回切換頁籤。
- 替代方案：保留獨立「我的訂單」TAB。未採用，因為下單與確認結果分離，互動不夠直覺。

### 3. 將共用提示統一收斂到 MessageBox state
- 決策：以單一 `messageBox` 狀態表示彈窗是否開啟、類型（success/error/warning/confirm）、標題、內容、確認動作與取消動作；原本 `message` / `error` 文字區塊改為透過 helper 開啟 MessageBox。
- 原因：A009 要求所有提示訊息都以 MessageBox 顯示，且刪除操作必須提供確認與取消按鈕，使用單一狀態模型最容易保證互動一致。
- 替代方案：保留既有 banner，僅在刪除時加入 confirm dialog。未採用，因為不符合 A009 「所有提示訊息均以 MessageBox 彈窗方式顯示」的要求。

### 4. 先做前端表單驗證警告，再進入 API 呼叫
- 決策：針對目前已知容易在前端先判斷的必要欄位與基本格式，先以 warning MessageBox 阻擋送出；其餘後端驗證失敗則以 error MessageBox 顯示 API 錯誤訊息。
- 原因：A009 把「表單驗證失敗」獨立為 warning 類型，因此前端需要有明確機制區分驗證警告與伺服器錯誤。
- 替代方案：所有驗證都交由後端處理，再統一顯示 error。未採用，因為這會失去 warning 類型的使用場景，也較不符合 UML sequence 的描述。

### 5. 刪除操作改為兩段式確認流程
- 決策：`submitDeleteErrorEmail`、`submitDeleteReportEmail`，以及員工取消訂單等破壞性操作，不直接綁在按鈕上執行，而是先開 confirm MessageBox；使用者按下確認後才實際呼叫 API，成功後再顯示 success MessageBox。
- 原因：A009 明確將刪除確認列為必要流程，且需要確認與取消兩個按鈕。
- 替代方案：使用瀏覽器 `window.confirm`。未採用，因為需求指定 MessageBox，且原生 confirm 無法與整體 UI 風格一致。

## Risks / Trade-offs

- [將所有提示集中到單一 MessageBox state，若狀態切換處理不完整，可能出現彈窗互相覆蓋或關閉時序錯亂] → 封裝 `openSuccessBox`、`openErrorBox`、`openWarningBox`、`openConfirmBox` helper，避免各 action 直接拼接狀態物件。
- [目前 `App.tsx` 體積已大，若直接把 TAB 與彈窗邏輯再塞進同檔案，可讀性會更差] → 至少抽離共用 `TabBar` / `MessageBox` 元件，必要時再把員工與管理員區塊拆成區域元件。
- [部分既有成功訊息發生在 API 後立即重載資料，若改成 MessageBox 可能影響畫面刷新時機] → 約定先更新資料，再開啟 success MessageBox，避免使用者確認後看到舊資料。
- [若 warning 條件定義不足，仍可能大量依賴後端錯誤訊息] → 第一版只先涵蓋明確必填與刪除確認流程，其餘沿用後端回傳錯誤，但保留後續擴充欄位驗證入口。

## Migration Plan

- 在前端新增 TAB 與 MessageBox 共用元件或等價抽象，先接入 `App.tsx`。
- 將登入區改為單一登入表單，移除入口切換按鈕與相依 state。
- 依角色建立頁籤定義與預設頁籤，將登入後原本同頁堆疊內容拆分成對應 tab panel。
- 將 `message` / `error` 文字提示改為 MessageBox helper 呼叫。
- 將刪除與取消類操作改為 confirm -> API -> success/error 的兩段式流程。
- 驗證員工與管理員主要操作在新頁籤結構下仍可完成；若需 rollback，可暫時保留舊版 inline banner 與非 tab 佈局分支，但 A009 完成後應移除。
