## Context

在目前前端資訊架構中，管理員已經有「員工管理」與「CSV 匯入」等獨立主 TAB，但「部門管理」仍留在「系統設定」中。這種切分有點尷尬，因為部門資料並不是低頻系統設定，而是實際支撐員工建立、員工篩選與資料維護的主檔。從使用者心智來看，部門管理更接近員工管理、供應商管理這類日常維運頁面，而不是 Email 設定一類的系統配置。

這次 change 的重點仍然是前端資訊架構與 OpenSpec 導覽規格調整，不涉及 A010 API 本身的功能擴充。也就是說，我們只是把既有部門維護內容搬到更合適的主導覽層級。

## Goals / Non-Goals

**Goals:**
- 在管理員主導覽新增「部門管理」主 TAB。
- 將部門清單、建立、修改內容從「系統設定」搬到新主 TAB。
- 讓「系統設定」回到純設定用途，不再承載部門主檔維護。
- 保持現有部門 API、單頁切換模式與 MessageBox 互動不變。

**Non-Goals:**
- 不新增部門刪除 API。
- 不改動 A010 的資料契約或後端 service / repository 行為。
- 不把部門管理再細分成更多子頁或多層路由。
- 不重新調整其他主 TAB 的預設 landing 規則。

## Decisions

### 1. 新增「部門管理」主 TAB，作為部門維護的唯一入口
- 決策：在管理員主導覽加入 `部門管理`，讓查詢、新增、修改部門都集中在這個主 TAB。
- 理由：A010 的部門維護已經是獨立能力，放在主 TAB 更符合功能權重與使用頻率。
- 替代方案：維持在系統設定中。未採用，因為這會延續目前層級不一致的問題。

### 2. 「部門管理」TAB 直接承載目前現有的單頁維護內容
- 決策：沿用現有部門建立表單與部門清單/編輯區塊，不新增額外次分頁。
- 理由：目前部門管理流程單純，單頁就足夠，不需要再增加一層切換。
- 替代方案：再加一層「部門列表 / 新增部門」子頁。未採用，因為會讓簡單流程過度分層。

### 3. 「系統設定」保留錯誤通知信箱等設定型內容
- 決策：系統設定移除部門管理後，聚焦於錯誤通知信箱與其他真正偏設定性的功能。
- 理由：這能讓主導覽各 TAB 的職責更清楚，避免系統設定變成雜項收納區。
- 替代方案：把所有主檔維護都放回系統設定。未採用，因為這與近期已確立的主 TAB 方向相反。

### 4. 管理員主 TAB 數量擴充為七個
- 決策：主導覽更新為「訂單管理」「供應商管理」「報表設定」「員工管理」「部門管理」「CSV 匯入」「系統設定」七個主 TAB。
- 理由：既然員工管理與部門管理都是獨立主檔維護能力，分成兩個主 TAB 比把其中之一藏在系統設定更一致。
- 替代方案：把員工管理與部門管理合併成單一主 TAB。暫不採用，因為目前 user request 是要把部門管理獨立成主 TAB。

## Risks / Trade-offs

- [主導覽再多一個 TAB 後，窄畫面更容易擁擠] → 實作時要確認 TabBar 的換行或橫向可用性。
- [若系統設定被收斂得太空，可能讓人覺得這個 TAB 功能太少] → 這是合理結果，代表設定和主檔維護已被清楚分開。
- [部門管理從系統設定搬出後，使用者可能預期 CSV 匯入中的 departments 區塊也要搬到同一頁] → 本次先不調整 CSV 匯入，因為那是 A012 的獨立能力，維持在 CSV 匯入主 TAB 內。

## Migration Plan

- 先更新 OpenSpec 主規格，將管理員主 TAB 擴充為含「部門管理」的版本。
- 在前端把部門建立/修改區塊搬到新的 `admin-departments` 主 TAB。
- 收斂「系統設定」內容，移除部門管理相關區塊。
- 驗證管理員仍可完成部門查詢、新增、修改，且不影響員工管理、CSV 匯入與系統設定既有流程。
