## Context

A002 已完成員工訂餐、供應商與菜單資料邊界，系統已經能產生隔日供應商所需的訂單資料，但目前仍缺少每天 17:00 自動彙整並通知供應商的執行機制。A003 要補上的不是新的人工操作頁，而是後端排程、訂單聚合、通知發送與異常追蹤能力，並且要與後續 A004 錯誤通知信箱設定銜接。

現有 backend 已有 Email service stub、orders / menus / suppliers 的資料模型，以及 A002 預留的 A003 邊界說明，因此 A003 會主要影響排程 service、通知內容組裝、通知記錄資料表與失敗補救流程。

## Goals / Non-Goals

**Goals:**
- 定義每日 17:00 自動執行的供應商通知排程。
- 定義隔日訂單查詢與依供應商、便當名稱彙整的資料邏輯。
- 定義供應商通知信與錯誤通知信的發送條件，以及 `notification_logs` 記錄格式。
- 保留與 A004 錯誤通知信箱來源、A005 月結報表與正式 SMTP 的擴充空間。

**Non-Goals:**
- 本 change 不提供新的人工管理頁面或手動重送 UI。
- 本 change 不定義 A004 錯誤通知信箱設定的 CRUD 規格，只先保留讀取介面邊界。
- 本 change 不處理供應商回覆、已讀回條或信件模板管理後台。
- 本 change 不處理月結統計或財務對帳，這屬於後續 A005。

## Decisions

### 1. 以固定排程驅動，而非人工 API 為主
- 決策：A003 的主流程以每日 17:00 scheduler 觸發為準，實作上可將核心通知流程抽成 service，讓後續若需手動重跑可重用相同邏輯。
- 原因：需求明確指定系統排程為唯一角色，排程應是主入口。
- 替代方案：先做手動 API、排程後補。未採用，因為會讓正式業務主流程延後。

### 2. 以「供應商 + 便當名稱」為彙整鍵
- 決策：查詢隔日 `orders` 後，需 join `menus` 與 `suppliers`，依供應商分組，再以便當名稱統計數量生成通知內容。
- 原因：這直接對應 A003 的通知需求，也和 A002 既有資料模型一致。
- 替代方案：只列出逐筆訂單。未採用，因為供應商需要的是聚合後的收單結果。

### 3. 通知結果一律寫入 `notification_logs`
- 決策：不論供應商通知成功、供應商通知失敗、發送異常，或排程本身出現系統錯誤，都寫入 `notification_logs`，並以 `status` 與 `error_message` 區分。
- 原因：這能保留稽核與重送基礎，也符合需求中的記錄要求。
- 替代方案：只記錄失敗案例。未採用，因為無法完整追蹤每日通知結果。

### 4. 錯誤通知與供應商通知分離
- 決策：供應商通知信與 A004 錯誤通知信箱通知使用不同組信邏輯與不同 recipient 集合；錯誤通知來源以獨立 provider 介面讀取，A003 先用 stub 或固定值實作。
- 原因：避免把正常通知與系統錯誤處理耦合在同一條發送邏輯中。
- 替代方案：所有通知共用同一發送流程與 recipients 判斷。未採用，因為會增加混淆與錯誤風險。

## Risks / Trade-offs

- [排程固定 17:00 觸發，若部署環境時區錯誤會導致提前或延後發送] → 統一使用 `Asia/Taipei`，並在 scheduler / log 中記錄 notify date。
- [若 Email service 發送部分成功、部分失敗，可能造成供應商已收到但錯誤通知未完整觸發] → 將每封通知獨立 try/catch，逐筆寫 log，再補發錯誤通知。
- [A004 尚未完成，錯誤通知收件來源暫時不穩定] → 先抽象成 provider 介面，A004 完成後直接替換資料來源。
- [隔日訂單查詢若資料量增長，join 與 grouping 成本會上升] → 初版先用後端聚合，後續可視流量改用查詢優化或預先彙整。

## Migration Plan

- 新增 `notification_logs` schema、model、repository 與通知 service。
- 新增 scheduler 與供應商通知流程，先以 stub Email service 驗證邏輯。
- 將 A002 現有訂單與供應商資料接入 A003 聚合流程。
- 若需 rollback，可暫時停用 scheduler bean，不影響 A001/A002 的核心訂餐流程。

## Open Questions

- A004 完成前，錯誤通知信箱是否先使用固定 demo recipients，或暫時只記錄 log 不實際發信。
- 若同一供應商隔日沒有任何訂單，是否完全不寄信，或寄送零訂單通知。
- 是否需要在 A003 就保留手動重跑的 internal endpoint，或延後到營運需求明確時再加入。
