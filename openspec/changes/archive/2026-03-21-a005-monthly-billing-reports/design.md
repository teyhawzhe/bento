## Context

目前系統已具備員工訂餐、供應商資料、菜單資料，以及 A003/A004 兩種通知邊界，但尚未有針對月結帳單的批次作業、記錄表與管理入口。A005 需要同時涵蓋排程、人工補送、報表內容彙整、Email 發送與歷史記錄查詢，會跨越 `orders`、`menus`、`suppliers`、後端排程服務與前端管理介面。

主要限制有三個：第一，月結範圍不是自然月，而是固定以上月 15 日到當月 14 日為一期；第二，收件者除了供應商通知信箱外，還包含 A008 報表收件信箱清單，但 A008 尚未實作；第三，發送失敗不能中斷整批流程，仍需處理剩餘供應商並保留可查詢的紀錄。

## Goals / Non-Goals

**Goals:**
- 定義每月 15 日自動執行的月結帳單批次流程。
- 提供管理員手動觸發與查詢月結發送記錄 API。
- 明確規範月結報表期間、依供應商與便當名稱的彙總規則，以及 TOTAL 計算方式。
- 定義 `monthly_billing_logs` 的資料粒度，讓每次實際發送都可追蹤成功或失敗。
- 為 A008 報表收件清單保留 provider 整合介面，避免 A005 未來重構收件來源。

**Non-Goals:**
- 不在 A005 內實作 A008 的收件信箱維護功能。
- 不在 A005 內導入 PDF、Excel 或附件檔案匯出，報表內容先以 Email 文字或 HTML 內容為主。
- 不處理對帳核准、付款狀態或會計系統串接。
- 不改變 A002 訂單建立與 A003 隔日通知的既有規則。

## Decisions

### 1. 月結期間以「執行日所屬月份的 15 日」為分界
- 決策：若排程或管理員在某月 15 日觸發，系統一律計算「上月 15 日 00:00:00 至當月 14 日 23:59:59」的訂餐資料。
- 原因：這與需求文件完全對齊，且能避免人工手動補送時因執行時間不同而產生不同報表區間。
- 替代方案：依觸發當下往前 30 天滾動計算。未採用，因為會讓同一期重跑結果不穩定。

### 2. 報表以供應商為主體彙整，並按便當名稱統計數量與小計
- 決策：使用 `orders JOIN menus JOIN suppliers` 取得資料，依 `supplier_id + menu_name + menu_price` 聚合數量與小計，再在每個供應商報表內計算 TOTAL。
- 原因：這與需求中的表格格式一致，也能直接反映實際請款內容。
- 替代方案：直接列出每筆訂單明細。未採用，因為不符合月結帳單彙總需求，且信件內容會過長。

### 3. `monthly_billing_logs` 以「每個實際收件地址」為一筆紀錄
- 決策：同一供應商同一期若同時寄給 `suppliers.email` 與多個 A008 收件者，系統對每個 `email_to` 各寫一筆 log。
- 原因：需求明確要求記錄實際發送地址；若只以供應商為一筆，無法區分哪一個收件者失敗。
- 替代方案：每個供應商每期只留一筆總結果。未採用，因為不利排錯與重送。

### 4. 手動觸發與排程共用同一個月結服務
- 決策：排程入口與 `POST /api/admin/reports/monthly` 都委派給同一個 application service，差異只在 `triggered_by` 是否為管理員 ID。
- 原因：可以確保排程與人工重跑的資料範圍、彙整與寫 log 邏輯一致。
- 替代方案：分成排程專用和 API 專用兩套流程。未採用，因為容易造成規則漂移。

### 5. A008 以 provider 介面保留整合點，A005 先明確依賴它
- 決策：A005 設計中定義報表額外收件者 provider，正式來源為 A008；在 A008 未落地前，實作可先提供空清單或 stub，但 contract 需固定。
- 原因：需求已指定 A008 為收件信箱來源，先建立介面邊界可降低未來整合成本。
- 替代方案：先把額外收件信箱硬編碼在 A005。未採用，因為之後仍需拆換且容易遺漏。

### 6. 發送失敗不中止整批，但必須完整記錄
- 決策：單一供應商或單一收件人發送失敗時，寫入 `failed` 與錯誤訊息後繼續處理後續資料；整批流程結束後由管理員透過查詢 API 檢視結果。
- 原因：這與需求「繼續處理下一廠商」一致，也能避免局部失敗阻斷整月作業。
- 替代方案：任一失敗即整批 rollback。未採用，因為批次 Email 無法真實回滾，且營運風險較高。

## Risks / Trade-offs

- [A008 尚未實作] → 先以 provider contract 固定依賴點，design 與 tasks 明確標示需接 A008 收件清單。
- [同一期手動重跑可能重複寄送] → 在 log 中保留 `billing_period_start/end` 與 `triggered_by`，讓管理員可辨識重跑紀錄；是否阻擋重跑留待後續決策。
- [歷史價格異動可能影響小計] → 若目前訂單僅連到 `menus.price`，需確認以當期菜單價格計算是否符合業務；必要時在實作階段補價格快照。
- [收件人數增加導致 log 筆數變多] → 以每個 `email_to` 一筆換取可追蹤性，查詢 API 需提供期間與供應商過濾能力降低閱讀成本。
