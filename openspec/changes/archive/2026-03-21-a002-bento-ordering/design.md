## Context

A001 已提供員工與管理員登入、JWT 角色授權，以及管理員維護員工帳號的基礎能力。A002 需要在這個基礎上加入系統真正的核心流程：管理員建立供應商與菜單，員工瀏覽下週工作日便當並在截止前完成訂餐，必要時由管理員在允許期限內取消指定員工訂單。

目前專案已有 React 前端與 Spring Boot 後端，但尚未建立 suppliers、menus、orders 的資料模型，也還沒有登入後的員工訂餐頁與管理員菜單頁。A002 同時受到 A003 的後續批次規則約束，因此需要先把資料邊界、截止邏輯與角色責任固定下來。

## Goals / Non-Goals

**Goals:**
- 定義供應商、菜單與訂單的資料邊界，以及菜單有效期間與訂餐日期規則。
- 讓員工可在登入後查看下週工作日可訂菜單、建立訂單、修改訂單、取消訂單與查看個人歷史訂單。
- 讓管理員可新增供應商、查詢含歷史的菜單清單、建立或編輯菜單，並取消指定員工訂單。
- 沿用 A001 的 JWT 角色授權，清楚分隔員工與管理員的操作介面與 API。
- 保留後續切換為 MySQL，並銜接 A003 供應商通知批次的擴充空間。

**Non-Goals:**
- 本 change 不處理採購發票、付款流程、對供應商的正式通知寄送與月結報表。
- 本 change 不定義管理員代替員工新增訂餐與查詢所有員工訂單，這屬於後續 A006。
- 本 change 不包含多租戶、SSO、LDAP 或其他身分整合。
- 本 change 不加入員工端價格顯示，員工端只看到可選便當資訊，不看到價格。

## Decisions

### 1. 菜單以單筆便當品項為主，並綁定供應商與有效期間
- 決策：`menus` 直接表示可被員工選取的便當品項，每筆資料包含 `supplier_id`、`name`、`category`、`description`、`price`、`valid_from`、`valid_to`。
- 原因：這與最新 A002 文件與 OpenAPI 一致，也能直接支援「查詢下週工作日可訂便當」與 A003 後續按供應商彙整。
- 替代方案：改成「一份菜單含多個餐點選項」的聚合模型。未採用，因為目前需求文件已把 `menus` 定義為單筆便當資料表。

### 2. 員工可訂範圍固定為下週工作日，截止規則採全域時間規則
- 決策：員工端 `GET /api/orders/menu` 僅回傳下週工作日且仍在 `valid_from` / `valid_to` 範圍內的便當；員工新增、更新、取消訂單均以「當週星期五 12:00 前」作為統一截止時間。
- 原因：需求文件明確指定「下週工作日」與固定截止規則，實作上也比每筆菜單各自設定截止更單純。
- 替代方案：每份菜單各自帶訂餐截止時間。未採用，因為這不符合目前 A002 的業務描述。

### 3. 同一員工同一日只保留一筆有效訂單，重複下單以最新一筆覆蓋
- 決策：`orders` 以 `employee_id + order_date` 作為業務唯一鍵；員工重複送出同日訂單時，後端更新既有記錄而非新增第二筆。
- 原因：這是 A002 已明定的訂餐規則，也能讓前端用最直接的「改單」心智模型操作。
- 替代方案：允許保留多筆歷史後再標記最後一筆有效。未採用，因為當前需求只要求保留一筆有效訂單。

### 4. 員工端與管理員端使用不同資料視圖
- 決策：菜單價格欄位保留於資料表與管理員 API，但員工端查詢菜單與個人訂單時不回傳價格欄位。
- 原因：A002 明確要求費用由公司吸收，且員工瀏覽便當選項時不顯示價格資訊。
- 替代方案：統一 API 回傳價格，再由前端隱藏。未採用，因為會讓員工端資料暴露超出需求。

### 5. 管理員取消指定員工訂單需銜接 A003 批次界線
- 決策：管理員取消員工訂單的最晚時間為該訂餐日前一日 17:00 前，並在設計上以 A003 批次執行前為同一條業務界線。
- 原因：需求已明確把這個截止時間與 A003 綁定，A002 必須先預留一致的判斷邏輯。
- 替代方案：沿用員工相同的星期五中午截止。未採用，因為管理員取消訂單的業務規則不同。

### 6. 角色導向的登入後入口沿用 A001 session
- 決策：前端沿用 A001 的 session 儲存方式；員工登入後進入個人訂餐頁，管理員登入後進入菜單與報表頁，所有 A002 API 按既有 JWT role 驗證。
- 原因：能重用既有認證邊界，避免在 A002 再次拆分登入模型。
- 替代方案：為 A002 重新建立 session 狀態或路由守衛模型。未採用，因為現階段沒有必要引入第二套授權流程。

## Risks / Trade-offs

- [固定以星期五 12:00 截止，若遇國定假日或補班可能不符實務] → 先依需求固定實作，將例外日曆規則留待後續 change 擴充。
- [員工端不回傳價格，若直接共用管理員 DTO 容易意外洩漏價格] → 分開定義員工端與管理員端回應模型。
- [管理員取消訂單截止時間依賴 A003 批次邊界，若未集中封裝容易出現規則漂移] → 將截止判斷抽成共用 service，供 A002 與 A003 共用。
- [同一員工同一日覆蓋訂單，若缺乏唯一性約束可能造成重複資料] → 在 repository 與資料表層加入業務唯一鍵與 upsert/覆蓋流程。

## Migration Plan

- 先新增 suppliers、menus、orders 的 OpenSpec 規格與任務，再進入實作。
- 實作時優先建立後端 domain model、repository 介面與截止規則 service，再串接前端員工訂餐頁與管理員菜單頁。
- 若專案仍使用記憶體 repository，可先以 in-memory 或 JDBC 並存方式落地，保持與 A001 的執行方式一致。
- rollout 時以「未登入不可見、已登入按角色顯示功能」方式漸進啟用；如需 rollback，可暫時關閉 A002 入口並保留 A001 登入能力。

## Open Questions

- `valid_from` / `valid_to` 是否只允許工作日，還是可先接受任意日期再由前端只篩出下週工作日。
- 管理員編輯菜單時，若該菜單已被員工選用，是否允許修改供應商與價格，仍待業務確認。
- 是否需要在 A002 就提供供應商清單查詢 API，或目前僅建立供應商即可。
