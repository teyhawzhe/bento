## 為什麼要做

公司員工訂便當系統需要先建立可靠的身分驗證入口，才能安全地承接後續訂餐、菜單管理與報表功能。A001 已定義員工與管理員的登入、密碼管理及帳號維護規則，現在需要將這些需求整理為可實作的 OpenSpec 變更基礎。

## 變更內容

- 新增員工登入能力，支援 `/login` 入口、JWT 發放、登入失敗提示、登出、忘記密碼與修改個人密碼。
- 新增管理員登入能力，支援 `/admin/login` 入口與 role=`admin` 的授權資訊。
- 新增管理員員工帳號管理能力，支援查詢員工、新增單筆員工、CSV 匯入、停用或啟用帳號，以及重設指定員工密碼。
- 建立前後端一致的 A001 API 契約、角色權限規則與基本 UI 流程。
- 補齊 A001 所需的最小資料模型與外部通知介面，作為後續串接 MySQL、Email 與 Docker 化部署的起點。

## 能力範圍

### 新增能力
- `employee-authentication`: 員工與管理員登入入口、JWT 角色授權、登出、忘記密碼與員工自行修改密碼。
- `employee-account-administration`: 管理員查詢員工、建立帳號、CSV 匯入、停用啟用與重設員工密碼。

### 調整中的既有能力
- 無。

## 影響範圍

- 受影響程式碼：frontend 登入與管理頁面、backend 驗證與員工管理模組、共用 API 型別與錯誤處理。
- 受影響 API：`/api/auth/*`、`/api/admin/auth/login`、`/api/admin/employees*`。
- 相依項目：JWT 套件、CSV 解析、Email 發送介面、後續資料庫與 Docker Compose 配置。
- 限制條件：不修改 `../uml` 內的需求文件與 UML 輸出，僅將其作為規格來源。
