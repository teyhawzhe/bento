## 背景

A001 是公司員工訂便當系統的第一個可執行能力，負責建立員工與管理員的登入入口、JWT 角色授權、密碼管理與管理員維護員工帳號流程。專案技術方向已在 `openspec/config.yaml` 定義為 React + Vite + Tailwind 前端，以及 Java 21 + Spring Boot 後端，後續會以 MySQL 與 Docker Compose 部署，但目前需求文件與 UML 位於 `../uml`，且不可直接修改。

目前尚未有正式資料庫 schema、SMTP 設定與部署腳本，因此這個 change 需要先定義一個可演進的 A001 邊界：先完成角色、API 契約、畫面流程與最小資料模型，並保留後續替換為 MySQL、正式 Email 與容器化部署的空間。

## 目標與非目標

**目標：**
- 定義員工與管理員兩個登入入口及對應 JWT role 規則。
- 定義忘記密碼、員工修改密碼與管理員重設密碼的驗證行為。
- 定義管理員查詢員工、新增員工、CSV 匯入、停用與啟用的需求邊界。
- 明確前後端責任分工，讓後續 A002 之後的功能可以直接依授權資訊擴充。
- 保持設計可在後續切換到 MySQL、SMTP 與 Docker Compose，而不必重寫需求規格。

**非目標：**
- 本 change 不定義 A002 之後的訂餐、菜單、報表與排程邏輯。
- 本 change 不要求修改 `../uml` 中的需求文件、OpenAPI 或 PlantUML。
- 本 change 不強制規範正式的雜湊演算法、SMTP 供應商或 JWT 黑名單儲存方案；這些可於實作或後續 change 中補強。
- 本 change 不包含 SSO、LDAP 或第三方身分提供者整合。

## 設計決策

### 1. 採用雙登入入口，但共用員工主資料
- 決策：員工使用 `/login`，管理員使用 `/admin/login`，後端依登入入口決定應發放的 role，並以 `employees` 為單一主資料表。
- 原因：這與 A001 需求及 UML 完全一致，也能避免重複的管理員資料模型。
- 替代方案：分開建立 `admins` 資料表。未採用，因為需求僅以 `is_admin` 區分權限，拆表會增加同步成本。

### 2. JWT 僅作為 A001 與後續模組的授權載體
- 決策：JWT payload 至少包含 `role`、使用者識別與基本追蹤資訊，後端受保護 API 必須驗證 role。
- 原因：A001 明確要求後端所有 API 依 JWT 的 `role` 欄位驗證權限，且 A002 後續功能也會依登入身分擴充。
- 替代方案：使用 server-side session。未採用，因為需求已指定 JWT。

### 3. 密碼流程以一致的密碼政策與通知介面處理
- 決策：忘記密碼、管理員重設密碼與員工修改密碼必須共用同一套密碼格式規則，並透過獨立通知介面發送信件。
- 原因：可避免不同流程出現不一致驗證，也方便未來替換 SMTP 或訊息佇列。
- 替代方案：各 API 各自驗證與發信。未採用，因為會增加重複實作與規則飄移風險。

### 4. CSV 匯入採逐列驗證、部分成功回報
- 決策：CSV 匯入不採整批交易失敗；每列獨立驗證，失敗列需回傳錯誤原因，成功列繼續建立帳號。
- 原因：這與 A001 流程定義一致，也更符合管理介面實務操作。
- 替代方案：任一列失敗即整批 rollback。未採用，因為不符合需求中的「跳過不匯入」描述。

### 5. 規格先以能力導向切分為兩份 spec
- 決策：將 A001 切成 `employee-authentication` 與 `employee-account-administration` 兩個 capability。
- 原因：登入與管理員維護雖屬同一需求編號，但測試關注點與 API 邊界不同，拆分後較容易在後續實作與驗證追蹤。
- 替代方案：全部寫成單一 capability。未採用，因為規模稍大，後續 archive 與差異追蹤會不清楚。

## 風險與取捨

- [JWT 僅靠 role 驗證，若缺乏過期與撤銷策略可能有安全風險] → 於實作中至少要求 expiration；後續再增加 refresh token 或黑名單方案。
- [先用記憶體或 mock 通知服務會與正式環境行為不同] → 在 design 與 tasks 中保留替換為 MySQL/SMTP 的介面邊界。
- [管理員與員工共用 employees 表，若未妥善限制入口可能造成權限混淆] → 規格要求登入入口與 role 驗證雙重檢查。
- [CSV 批次匯入若缺乏格式與重複驗證，會造成資料品質問題] → 規格要求欄位完整性、帳號/Email 唯一性與錯誤明細回傳。

## 佈署與演進規劃

- 先建立 A001 的 proposal、specs、design 與 tasks，作為正式實作的唯一依據。
- 實作階段先交付可跑通的前後端流程與最小資料模型。
- 後續在不改變 A001 規格的前提下，將 repository 替換為 MySQL、通知服務替換為 SMTP，並加入 Docker Compose。
- 若部署後需 rollback，可回退到未啟用 A001 的版本；因本 change 主要新增能力，不涉及既有線上資料遷移。

## 待確認事項

- 正式環境的 JWT secret 與配置管理要由 `.env`、Vault 或部署平台變數提供，仍待專案部署策略確認。
- SMTP 或通知服務的正式供應商尚未指定。
- 前端 routing 是否在下一個 change 中導入 React Router，或維持較輕量的入口頁模式，仍可依專案規模再決定。
