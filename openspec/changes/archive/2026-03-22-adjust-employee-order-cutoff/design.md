## Context

A002 最新文件仍保留員工以星期五 12:00 作為統一截止時間，但把可訂日期的定義改成「截止日後一天到下週五」的固定區間，並明確允許週末若有有效菜單也可訂。先前實作把規則誤解成依每個訂餐日各自計算前一工作日 17:00，導致員工會看到錯誤的日期範圍，例如 2026-03-23 仍可被選取，而不是直接顯示 2026-03-28 到 2026-04-03 的區間。

這次 change 需要把 backend deadline service、訂單驗證、員工菜單查詢與前端提示一起拉回 A002 的正式版本，另外也要把管理員取消員工訂單的截止時間同步改為前一日 16:30。

## Goals / Non-Goals

**Goals:**
- 讓員工只看到「本次星期五截止後到下週五」這個固定訂購區間，而不是錯誤的滾動工作日視窗。
- 讓員工建立、更新、取消訂單都統一受本次星期五 12:00 截止控制。
- 允許週末若有有效菜單時也能成為可訂日期。
- 把管理員取消指定員工訂單的截止時間改為前一日 16:30，與 A002 文件一致。

**Non-Goals:**
- 本 change 不處理國定假日、補班日或自訂行事曆。
- 本 change 不調整價格顯示、供應商資料模型或歷史訂單查詢格式。

## Decisions

### 1. 員工可訂視窗改為「本次截止日後一天到下週五」
- 決策：員工查詢菜單時，系統以當前訂購週期的星期五 12:00 為基準，回傳從該截止日後一天開始，到下週五為止的日期區間；只要菜單在該日有效，週六與週日也可列入。
- 原因：這完全對應 A002 新版文件與你確認的範例，像 2026-03-23 應顯示 2026-03-28 到 2026-04-03。
- 替代方案：依目標訂餐日逐日滾動。未採用，因為這正是上一版錯誤實作的來源。

### 2. 員工新增/修改與取消使用不同截止規則
- 決策：員工新增、更新訂單仍使用本次星期五 12:00 作為統一截止；員工取消自己的訂單則改為該訂餐日前一日 16:30。
- 原因：A002 最新文件已明確拆開這兩條規則，且要求前後端都要驗證取消的 `16:30` 邊界。
- 替代方案：讓取消也沿用星期五 12:00。未採用，因為與最新 A002 文件不符。

### 3. 可訂日期與截止判斷集中在共用 deadline service
- 決策：由 backend deadline service 提供「本次員工截止時間」、「區間起點」、「區間終點」與日期清單方法，`GET /api/orders/menu` 與建立、更新、取消 API 全部共用。
- 原因：避免前端、controller、service 各自重算，導致邊界時間與錯誤訊息不一致。
- 替代方案：在各 API 端點各自檢查。未採用，因為後續再調整規則時維護成本高且容易遺漏。

### 4. 取消按鈕由前端依每筆訂單截止時間決定是否顯示
- 決策：員工端「我的訂單」區塊在渲染每筆訂單時，依本機時間與該訂單前一日 16:30 比較；逾期則隱藏取消按鈕，點擊取消前再檢查一次。
- 原因：A002 明確要求前端載入記錄時就要隱藏逾期訂單的取消按鈕，避免使用者誤以為還能取消。
- 替代方案：保留按鈕、只在按下時顯示錯誤。未採用，因為這不符合文件要求。

### 5. 管理員取消訂單的截止時間改為前一日 16:30
- 決策：`ensureAdminCancellationWindowOpen` 使用前一日 16:30；管理員代訂建立的 17:00 邏輯暫不變動。
- 原因：A002 最新文件只明確更新取消截止時間，且你目前先處理 A002 邏輯。
- 替代方案：連同管理員代訂一併調整。未採用，因為那會碰到 A006 邏輯，超出這一步範圍。

## Risks / Trade-offs

- [若截止時間切換點處理錯誤，星期五中午後可能仍看到上一個週期日期] → 以單一 `employeeOrderDeadline()` 作為區間基準，並補星期五中午邊界測試。
- [前端若沿用舊文案，使用者仍會以為取消和新增共用同一個截止] → 將頁面提示與訂單卡片一併標示每筆訂單的取消截止時間。
- [週末日期現在可出現在清單，若菜單查詢仍只看工作日會造成空洞] → 由後端直接回傳實際有有效菜單的日期清單，前端只顯示後端給的結果。
- [A006 之後若也調整管理員代訂截止，可能還要再拆一次規則] → 目前只分離取消與建立兩種管理員截止邏輯，降低後續修改成本。

## Migration Plan

- 先更新 `employee-bento-ordering` delta spec，明確把員工規則改回星期五 12:00 與固定日期區間。
- 實作時先調整 backend deadline service、`OrderService` 驗證與管理員取消截止，再更新員工菜單查詢與前端提示。
- rollout 後若需回滾，可恢復舊的固定「下週工作日」查詢方式；資料表與 API 結構不需要變更。
