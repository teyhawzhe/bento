## Context

本需求延伸既有 `bento-supplier-management` 與 `bento-menu-management` 能力，目標是在供應商管理畫面內集中完成兩件事：選擇供應商，以及建立與查看該供應商的菜單。操作核心是單一供應商選擇器，該狀態同時驅動建立菜單表單與菜單清單，避免使用者在不同區塊輸入不一致的 supplier id。

## Goals / Non-Goals

**Goals:**
- 在供應商管理畫面提供獨立的建立菜單 tab。
- 使用下拉式供應商選擇器作為該 tab 的主要上下文。
- 選定供應商後自動帶入新建菜單所需的 supplier id，減少手動輸入。
- 讓菜單清單依目前選取的供應商過濾，只顯示相關菜單。
- 明確處理未選供應商時不可送出的狀態與提示。

**Non-Goals:**
- 不變更供應商主資料欄位與供應商 CRUD 規則。
- 不重新設計整體供應商管理資訊架構，只新增必要 tab 與互動。
- 不在本需求中新增跨多供應商批次建立或批次轉移菜單能力。

## Decisions

### 1. 在供應商管理畫面新增獨立菜單 tab
- 供應商管理頁新增一個專門處理菜單的 tab，與供應商基本資料管理分開，避免單頁混雜過多表單。
- tab 進入後預設顯示供應商選擇器、建立菜單表單與菜單清單區塊。

### 2. 以單一供應商選擇器作為頁面上下文
- 畫面頂部提供供應商下拉式選項，內容來自既有供應商清單資料。
- 使用者一旦選取供應商，前端保留 `selectedSupplierId` 與必要的顯示名稱，供表單與清單共用。
- 若尚未選擇供應商，建立菜單送出按鈕應維持不可用，並提示需先選擇供應商。

### 3. 建立菜單表單自動帶入 supplier id
- 新建菜單時不讓使用者手動輸入 supplier id，而是由目前下拉選取值自動帶入 request payload。
- 若切換供應商，後續新建菜單動作需使用新的 supplier id。
- 若編輯現有欄位時表單已輸入其他內容，切換供應商不應默默送出；只更新 supplier id 上下文，待使用者自行確認後送出。

### 4. 菜單清單依供應商篩選
- 菜單清單與建立表單共用同一個供應商選擇器。
- 選定供應商後，畫面只載入並顯示該供應商的菜單；切換供應商時重新查詢。
- 若未選供應商，可採空狀態提示，避免預設顯示所有菜單造成上下文混亂。
- 若現有 API 已支援 supplier id 篩選，沿用既有參數；若未支援，新增明確的 supplier id 查詢條件。

## Risks / Trade-offs

- 若供應商數量很多，下拉選單可能需要後續再補搜尋能力；本次先以可選取為主。
- 切換供應商時若表單已有未送出的內容，使用者可能誤以為資料仍屬於原供應商，因此 UI 需要明確反映目前所屬供應商。
- 若菜單 API 目前未提供 supplier id 篩選，後端需補 API 契約與測試。

## Migration Plan

- 先新增供應商管理畫面的 tab 與供應商選擇狀態。
- 再串接建立菜單表單，改由所選供應商帶入 supplier id。
- 最後讓菜單清單依所選供應商查詢，並補齊空狀態與切換流程驗證。

## Open Questions

- 下拉選單是否只顯示啟用中的供應商，或需包含停用供應商，待實作時確認既有業務規則。
- 若使用者切換供應商時表單已有未儲存內容，是否需要額外警示，可在實作時依現有 UX 模式決定。
