## Context

A012 要求管理後台新增一個獨立的 CSV 匯入主 TAB，並與「系統設定」維持同一層級，集中管理員工、部門、供應商與便當菜單四種主檔/業務資料的範本下載與批次匯入。`../uml/openapi.yaml` 已定義一組專屬 A012 API：一支 `GET /api/admin/import/template/{type}` 負責下載範本，四支 `POST /api/admin/import/*` 負責上傳 CSV 並回傳成功匯入資料清單或匯入錯誤結果。

目前專案只存在較早期的員工 CSV 匯入能力，且尚未收斂成 A012 的統一模型；部門、供應商與菜單也都沒有對應的 CSV 匯入入口。若直接各自補 API，容易讓表頭驗證、批次大小、rollback 行為與回傳格式不一致，因此這次設計重點是先固定一個共用匯入框架，再讓四種資料型別各自提供欄位映射與重複檢查邏輯。

## Goals / Non-Goals

**Goals:**
- 提供四種 CSV 範本下載與批次匯入 API，契約對齊 A012 與 OpenAPI。
- 統一支援 UTF-8 驗證、表頭完全比對、必填欄位檢查、批次內重複檢查與既有資料衝突檢查。
- 明確定義每 5000 筆為一個 transaction 批次，批次失敗時 rollback 並回傳處理到的 CSV 行號。
- 在管理後台新增與「系統設定」同層級的 CSV 匯入主 TAB，集中呈現下載、上傳與成功匯入資料清單。
- 讓員工、部門、供應商、菜單匯入成功後的資料模型可直接映射既有主檔查詢結果。

**Non-Goals:**
- 不新增資料表或 CSV 匯入歷史紀錄表。
- 不改寫既有單筆建立/修改 API 的業務規則。
- 不在 A012 內加入背景工作、非同步匯入或檔案佇列。
- 不在第一版支援 ZIP、Excel 或非 CSV 格式。

## Decisions

### 1. 建立共用 CSV import service 框架，四種型別各自提供 schema 與 row mapper
- 理由：A012 的四種匯入流程在編碼驗證、表頭驗證、批次 transaction、錯誤回傳格式上高度一致，只在欄位定義、重複檢查與資料寫入目標不同。用共用框架可以避免四份幾乎相同的匯入程式分岐。
- 替代方案：每種型別各自獨立實作完整匯入流程。未採用，因為會增加維護成本，也更難確保四種匯入行為一致。

### 2. 範本下載採動態字串/檔案回傳，不落地存實體 template 檔
- 理由：A012 的範本內容只有固定表頭，直接由後端依 `type` 回傳 `text/csv` 最簡單，也能避免模板檔與 OpenAPI/需求文件脫節。
- 替代方案：把範本 CSV 檔放在資源目錄提供下載。未採用，因為每次欄位異動都要同步調整靜態檔案。

### 3. 匯入錯誤分為兩層：格式錯誤 `400`，資料驗證/交易失敗 `422`
- 理由：OpenAPI 已把表頭格式不正確歸在 `400`，把資料驗證失敗且已 rollback 的情境歸在 `422`。這能讓前端清楚區分「檔案本身不合法」與「資料內容需修正後重傳」。
- 替代方案：所有匯入失敗都回 `400`。未採用，因為會失去錯誤類型語意，也不符合 OpenAPI。

### 4. 同批次重複檢查在記憶體內先完成，再進 transaction 寫入
- 理由：A012 明確要求同批次內重複資料也要被拒絕。先在記憶體內完成 batch-level duplicate detection，可在寫入前就回報更明確的行號與原因，並降低半途失敗風險。
- 替代方案：完全依賴資料庫唯一鍵。未採用，因為資料庫例外不容易對應到精確 CSV 行號，也較難呈現符合 A012 的錯誤訊息。

### 5. CSV 匯入作為管理員主 TAB，與系統設定同層級
- 理由：CSV 匯入涵蓋四種資料模型、範本下載與批次結果清單，資訊量與操作密度都明顯高於一般系統設定項。把它放在與「系統設定」同層級的主 TAB，更符合使用頻率與資訊架構，也能避免系統設定子頁過度擁擠。
- 替代方案：將 CSV 匯入維持在「系統設定」子頁籤。未採用，因為會讓「系統設定」同時承載錯誤通知信箱、員工帳號工具、部門管理與四種 CSV 匯入，層級過深且不利於快速進入。

### 6. 前端在 CSV 匯入主 TAB 內以「資料類型卡片」呈現四種範本與上傳入口
- 理由：A012 要求集中管理四種匯入流程，但不要求四個完全分離頁面。以四張卡片或四個區塊並列，可在同一頁清楚展示範本、欄位摘要、檔案選擇與匯入結果。
- 替代方案：每種資料類型做成獨立二級路由或子頁。未採用，因為目前前端是單頁 tab shell，維持同頁分區成本最低。

### 7. 匯入成功後回傳「本次成功匯入資料」而非只回 success count
- 理由：A012 明確要求成功後可查看已匯入的資料清單，OpenAPI 也為四種類型定義了 `Import*Result` schema。後端應直接回傳成功資料陣列，前端再渲染清單。
- 替代方案：僅回成功筆數，讓前端額外重查列表。未採用，因為會增加 API 往返，且不符合 A012 的直接回傳成功資料語意。

## Risks / Trade-offs

- [A012 會同時碰到四種資料模型與前後端 tab shell，範圍比單一 CRUD change 大] → tasks 必須拆成共用匯入框架、四種型別 adapter、前端頁面三大區塊，避免一次做成大泥球。
- [5000 筆 batch transaction 需要精準回報失敗行號，若交易內才發現錯誤，錯誤定位會變差] → 盡量把表頭、必填、格式與同批次重複檢查前置，保留 DB 約束錯誤作為最後防線。
- [菜單匯入依賴 `supplier_id` 與日期/價格格式，最容易出現跨表與格式驗證] → 在菜單 import adapter 中明確拆出供應商存在檢查、日期範圍檢查與 `(supplier_id, name)` 唯一性檢查。
- [員工匯入目前已有舊流程，A012 可能與既有欄位或回傳格式衝突] → 明確以 A012 / OpenAPI 欄位 `username,name,email,department_id` 為準，必要時淘汰或改造舊匯入入口。
- [前端若一次在同頁顯示四種大型結果清單，畫面可能過長] → 第一版僅顯示最近一次匯入結果，依資料型別分區呈現，不做歷史保留。

## Migration Plan

- 先補 OpenSpec delta specs，明確 A012 與既有 employee/department/supplier/menu capabilities 的交界。
- 後端新增 A012 import controller 與共用 import service，先實作 template download 與員工/部門匯入，再擴充供應商/菜單。
- 前端在管理員 tab shell 中加入與「系統設定」同層級的 CSV 匯入主 TAB，串接範本下載與四種上傳流程。
- 補上 controller/service 測試與前端驗證，確保格式錯誤、資料錯誤、成功匯入三類回應都可正確顯示。

## Open Questions

- `ImportErrorResponse` 的實際欄位命名是否已在 OpenAPI components 完整定義：目前設計假設至少包含錯誤訊息與失敗行號，實作前需再對照 schema 細節。
- 菜單匯入的 `category` 是否需限制在既有 enum 範圍：A012 文本未展開，但實作時應以既有菜單建立 API 的欄位規則為準。
