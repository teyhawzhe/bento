## Context

`../uml/input/A002.md` 最新版已把員工訂餐規則細化成三條不同邊界：員工新增/修改以本週星期五 12:00 為截止、員工取消以各訂餐日前一日 16:30 為截止、管理員取消指定員工訂單也以各訂餐日前一日 16:30 為截止。另一方面，可訂日期不再只是「下週工作日」的舊描述，而是改成「截止日後到下下週五前，且需落在菜單有效期間內」，並明確允許週末若有有效菜單也可訂。

目前 OpenSpec 主規格只約束了部分 envelope 與欄位契約，尚未完整表達這些截止規則與前端互動責任；另外，現有程式與 README 仍保留 `GET /api/menus?include_history=` 與「顯示歷史菜單」語意，但最新版 A002 已退回成單純「查詢菜單清單」。這輪全量比對也發現另一個相鄰殘留：A010 規格與後端都未提供部門刪除 API，但前端仍保留 `deleteDepartment` helper。這些都說明專案存在少量「舊行為殘留於實作 / 文件，但不再屬於正式契約」的情況，因此需要先用設計文件統一名詞與邊界，避免後端驗證、前端畫面與後續測試各自解讀。

## Goals / Non-Goals

**Goals:**
- 把員工新增/修改、員工取消、管理員取消三種截止規則拆開定義，避免沿用同一個錯誤時間邏輯。
- 固定 `GET /api/orders/menu` 的可訂日期視窗，讓前端與後端都依相同日期區間提供與驗證菜單。
- 明確前端在取消流程中的責任：列表載入先隱藏逾期取消按鈕，送出前再做一次時間檢查；後端仍保留最終授權。
- 對齊 `GET /api/menus` 到最新版 A002，只保留菜單清單查詢本身，不再把歷史菜單切換視為 A002 的必要能力。
- 把本次規格審計發現的非契約殘留一併整理，例如前端未使用但與 A010 規格不一致的部門刪除 helper，避免之後再次混淆為正式需求。
- 讓 OpenSpec delta specs 能直接映射 `../uml/input/A002.md` 與 `../uml/openapi.yaml`，作為後續實作與驗收依據。

**Non-Goals:**
- 本 change 不新增 A002 以外的新業務流程，例如管理員代替員工新增訂單、月結報表或供應商通知寄送。
- 本 change 不調整供應商或菜單資料模型欄位。
- 本 change 不改動 `../uml` 文件內容，只對齊到其現況。

## Decisions

### 1. 將截止判斷拆成三種明確規則，而不是單一通用截止時間
- 決策：員工新增/修改訂單使用「本週星期五 12:00」；員工取消與管理員取消都使用「訂餐日前一日 16:30」。
- 原因：最新版 A002 已清楚區分這三種行為，若仍共用單一截止規則，會讓取消操作與建立/修改操作互相污染。
- 替代方案：所有 A002 訂單行為都統一以星期五 12:00 判斷。未採用，因為與最新版文件不符，也無法支援逐日取消邊界。

### 2. 可訂菜單視窗以「下一個星期五 12:00 截止後的隔日，到再下一個星期五」為業務區間
- 決策：員工端可訂日期由一個固定期間決定，區間起點為本次截止日後一天，終點為下下週五；實際可見選項還要再與菜單 `valid_from` / `valid_to` 交集。
- 原因：這是 A002 新版文件與 UML 活動圖已經描述的行為，也能覆蓋週末若有有效菜單即可訂的要求。
- 替代方案：維持舊版「只顯示下週工作日」。未採用，因為會漏掉週末有效菜單，且與最新版文字與範例不一致。

### 3. 前端與後端共同承擔取消截止驗證，但後端是最終仲裁
- 決策：前端在載入個人訂單或管理員訂單清單時，先依各筆 `order_date` 決定是否顯示取消按鈕；真正送出刪除前再檢查一次當前時間；後端在 `DELETE` API 內進行最終 16:30 驗證並於逾期時回傳 403。
- 原因：這可同時滿足 A002 對使用者體驗與資料一致性的要求，避免頁面停留過久後仍送出過期操作。
- 替代方案：只做後端驗證，不處理按鈕顯示。未採用，因為不符合新版 A002 對前端流程的明確要求。

### 4. 本次以修改既有 capabilities 的 delta specs 呈現，不新增新的 capability
- 決策：將變更收斂在 `employee-bento-ordering`、`admin-order-management` 與 `bento-menu-management` 三個既有能力下，新增或修改 requirement/scenario 來描述更新後的行為。
- 原因：這些差異屬於既有流程的規則更新，不是新模組；用 delta specs 最能保持主規格語意清楚。
- 替代方案：新增獨立 capability 例如 `order-deadline-policy`。未採用，因為會把同一業務流程切得過細，增加理解成本。

### 5. 將「歷史菜單」視為超出最新版 A002 範圍的延伸能力
- 決策：A002 對 `GET /api/menus` 只保留基本菜單清單查詢與編輯流程，不再要求 `include_history` 查詢參數，也不把「顯示歷史菜單」作為 A002 驗收條件。
- 原因：最新版 `../uml/input/A002.md` 與 `../uml/openapi.yaml` 只保留「查詢菜單清單（管理員）」；若繼續把歷史模式綁在 A002，會讓規格與實作現況持續分岐。
- 替代方案：保留歷史查詢，但視為隱性兼容行為。未採用，因為會讓 A002 邊界不清楚，且不利於後續驗收與文件維護。

### 6. 對於不屬於任何現行契約的殘留 helper，應視為清理項而非新需求
- 決策：像前端 `deleteDepartment` 這類不在 `../uml/input/A010.md`、OpenAPI 與後端 controller 契約內的 helper，應在本次實作時視為殘留清理，而不是保留成隱性能力。
- 原因：這類殘留雖然未必已被畫面使用，但會誤導團隊以為系統支援規格外 API，長期會讓 README、OpenSpec 與程式碼認知分裂。
- 替代方案：先忽略這些殘留，僅處理 A002 主流程。未採用，因為本次已完成跨規格審計，順手清理能降低後續再次誤判的成本。

## Risks / Trade-offs

- [可訂日期區間文字容易被誤解為固定顯示工作日] → 在 spec scenario 中直接寫出「允許週末有效菜單」與具體日期範例。
- [前端與後端各自計算 16:30，若時區或邊界實作不同會產生不一致] → 要求實作共用同一套時區假設與 deadline helper/service，並在測試中覆蓋邊界時間。
- [若直接移除 `include_history`，可能影響既有管理員操作習慣或其他 change 的隱性依賴] → 先在 spec 與 tasks 中明確界定 A002 範圍，再於實作時檢查 README、前端頁面與後端 API 是否需要同步清理或保留兼容。
- [若清理規格外 helper 時未確認實際引用情況，可能誤刪尚未顯示但已串接的流程] → 實作前先盤點引用，再只移除未使用且不屬於契約的 helper / 文案。
- [只更新 OpenSpec 而未同步程式碼時，現況行為仍可能與規格不符] → tasks 明確拆出前端、後端、測試三條實作路徑。

## Migration Plan

- 先以 delta specs 更新 `employee-bento-ordering`、`admin-order-management` 與 `bento-menu-management` 的 requirement 與 scenarios。
- 之後實作時先集中整理 deadline 計算與日期視窗邏輯，再分別調整 backend API 驗證、菜單查詢參數與 frontend 顯示行為。
- 若實作過程需要回滾，可先保留既有 API 與畫面，只撤回新的日期視窗與取消截止判斷，不影響既有登入與查詢骨架。

## Open Questions

- `GET /api/orders/menu` 的回應是否需要額外回傳每個日期的 `cancelable` 或 `deadline` 類輔助欄位；目前 A002 未要求，本 change 先不新增。
- 若菜單有效區間跨越新版可訂區間邊界，是否需要在前端顯示「因截止規則不可訂」的說明；目前先聚焦規則正確，不擴充 UI 文案細節。
- A010 前端殘留的 `deleteDepartment` helper 是否完全未被任何畫面或測試引用；若是，實作階段可直接移除。
