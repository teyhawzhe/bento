## Context

目前員工前端的主 TAB 只有「訂便當」與「我的訂單」，而修改密碼被塞在「我的訂單」頁面右側。這種設計 technically 還有功能，但 discoverability 很差，使用者很容易以為修改密碼功能已經消失。另一方面，A001 的最新需求也明確指出系統管理員同樣要有修改自己的密碼頁面，但目前管理員前端缺少明確入口。

這次 change 不是擴充認證邏輯，也不是新增 API，而是把已存在的能力整理到更符合 A001 的導覽結構中。

## Goals / Non-Goals

**Goals:**
- 將員工端的修改密碼提升成獨立主 TAB。
- 讓管理員端在系統設定中有清楚可見的修改自己密碼入口。
- 保持現有修改密碼成功後重新登入、refresh token 作廢與 MessageBox 行為不變。
- 降低「功能存在但使用者找不到」的情況。

**Non-Goals:**
- 不新增新的修改密碼 API。
- 不改變密碼政策或 email 通知規則。
- 不把管理員修改密碼提升成新的管理員主 TAB。
- 不重做登入頁或認證頁的整體版面。

## Decisions

### 1. 員工端「修改密碼」升成獨立主 TAB
- 決策：員工主導覽從目前兩個主 TAB 擴充為「訂便當」「我的訂單」「修改密碼」三個主 TAB。
- 理由：A001 明確要求員工登入後有修改密碼頁面，把它做成獨立主 TAB 最容易被發現，也能避免和訂單資訊混在一起。
- 替代方案：保留在「我的訂單」頁內。未採用，因為實際上已經造成使用者誤判功能消失。

### 2. 管理員端修改自己密碼先放在系統設定
- 決策：在「系統設定」中加入獨立而清楚的修改自己密碼區塊，不再新增管理員主 TAB。
- 理由：管理員主導覽目前已經有七個主 TAB，再加一個密碼 TAB 會讓導航過長；而修改自己密碼屬於個人設定，放在系統設定是合理位置。
- 替代方案：管理員新增第八個主 TAB「修改密碼」。暫不採用，除非後續你希望把所有個人設定都提升為主導覽。

### 3. 員工「我的訂單」頁只保留訂單內容
- 決策：把員工「我的訂單」頁中的修改密碼表單移除，讓這個頁面聚焦在訂單記錄與取消操作。
- 理由：一個頁面只處理單一主要任務，可降低認知負擔，也讓頁面內容更一致。
- 替代方案：保留雙入口，同時在「我的訂單」頁與新 TAB 都放修改密碼。未採用，因為容易造成重複維護與混亂。

## Risks / Trade-offs

- [員工主 TAB 增加到三個後，文案與導覽說明需要同步更新] → 實作時要一起更新 spec 與畫面提示文字。
- [管理員如果預期修改密碼也會像員工一樣是主 TAB，可能仍覺得入口不夠顯眼] → 系統設定中的密碼區塊要放在夠清楚的位置與標題。
- [員工與管理員使用不同入口層級，可能讓規格看起來不完全對稱] → 這是刻意的 UX 取捨，因為兩邊導覽密度不同。

## Migration Plan

- 先更新 OpenSpec 導覽與認證規格，固定員工三個主 TAB 與管理員系統設定密碼入口。
- 在前端將員工修改密碼區塊從「我的訂單」頁抽離成 `employee-change-password` panel。
- 在管理員系統設定加入修改自己密碼區塊，串接管理員密碼修改 API。
- 驗證員工與管理員都可成功修改密碼，且成功後仍會要求重新登入。
