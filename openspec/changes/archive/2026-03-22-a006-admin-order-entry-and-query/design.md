## Context

目前 `admin-order-management` 只涵蓋管理員在截止前取消指定員工訂單，尚未提供管理員查詢全員訂餐紀錄與代為補單的入口。A006 會同時跨越前端管理頁、後端管理 API、`orders` 資料模型與 A003 17:00 截止規則，因此需要先固定設計邊界，避免後續在截止時間、資料欄位與查詢結果上出現前後端不一致。

既有系統已具備員工、菜單、訂單與 JWT/角色驗證能力，A006 應優先重用 A002 的訂單建立規則與 A003 的每日通知時點，只在管理員路徑增加「查詢所有員工」與「代替員工新增隔日訂單」兩個能力。需求文件已指定 `orders.created_by` 要記錄實際操作管理員，因此本次也包含資料庫 migration 與稽核欄位寫入策略。

## Goals / Non-Goals

**Goals:**
- 定義管理員查詢全員訂單的 API、篩選條件與回傳欄位。
- 定義管理員代替員工建立隔日訂單的驗證流程，並與 A003 的 17:00 截止規則一致。
- 補上 `orders.created_by` 的資料模型、寫入規則與查詢使用方式。
- 讓前端管理頁可以在同一個訂單總覽流程完成查詢、選員工、選日期與代訂。

**Non-Goals:**
- 不改變一般員工既有的訂餐截止規則與個人訂餐流程。
- 不在 A006 內處理管理員修改或取消代訂以外的其他批次作業。
- 不重構 A002 既有訂單資料模型中與 A006 無關的欄位或 API 命名。

## Decisions

### 1. 管理員查詢與代訂共用 `admin-order-management` 模組
- 決策：沿用既有管理員訂單 capability，在 backend 內由同一組 controller/service 負責 `GET /api/admin/orders`、`POST /api/admin/orders` 與既有取消能力。
- 原因：查詢、代訂、取消都依賴相同的管理員權限、訂單資料與截止規則，集中在同一模組可避免規則分散。
- 替代方案：拆成新的 capability 或獨立 controller。未採用，因為會讓 A006 與 A002/A003 的管理規則重複散落。

### 2. `POST /api/admin/orders` 只允許建立隔日訂單
- 決策：管理員代訂 API 僅接受「明日」的訂餐日期，並在 server 端重新計算是否早於該訂餐日前一日 17:00。
- 原因：需求明確描述為新增隔日訂便當，且截止時間與 A003 批次通知前完成直接綁定，限制日期可降低誤用。
- 替代方案：允許管理員指定任意未來日期。未採用，因為超出需求範圍，也會讓截止與菜單可見規則變複雜。

### 3. 代訂沿用既有訂單建立驗證，另外寫入 `created_by`
- 決策：管理員代訂時重用 A002 建立訂單的菜單有效性、員工存在性、重複訂單檢查等規則，只在成功寫入時將 `created_by` 設為管理員 `employee.id`。
- 原因：可確保管理員代訂與員工自行訂餐在資料一致性上沒有分歧，同時滿足稽核需求。
- 替代方案：建立管理員專用訂單表或只在 log 記錄操作者。未採用，因為會增加查詢與通知整合成本，且需求已指定欄位落在 `orders`。

### 4. 管理查詢 API 直接回傳展示所需的 JOIN 欄位
- 決策：`GET /api/admin/orders` 以訂單為主查詢，支援 `date`、`employee_id` 篩選，回傳至少包含訂單日期、員工 ID/姓名、便當名稱、價格、供應商名稱、建立者資訊。
- 原因：需求明確要求總覽頁可直接顯示員工姓名與便當名稱，若前端需要二次查詢員工或菜單會增加頁面複雜度。
- 替代方案：只回傳訂單與外鍵 ID，前端再自行組裝。未採用，因為資料關聯已在後端最清楚，也不利於篩選後列表效能。

### 5. `created_by` 允許員工自助訂單為空值
- 決策：migration 新增 `orders.created_by` 可為 nullable foreign key；員工自行建立的既有資料與後續資料可維持 `NULL`，管理員代訂才寫入管理員 ID。
- 原因：既有訂單資料沒有此欄位，直接要求非空會讓歷史資料補值困難，且需求只強制代訂場景需記錄操作者。
- 替代方案：將員工自助訂餐時 `created_by` 也填入本人 ID。未採用，因為不在本次需求範圍，也會改變既有資料語意。

## Risks / Trade-offs

- [截止時間與時區判斷不一致] → 由 backend 單一服務統一以系統時區判斷「訂餐日前一日 17:00」，前端只顯示提示文字，不自行決定是否可送出。
- [既有 `orders` 資料表新增外鍵可能影響 migration] → 採 nullable 欄位與漸進式 migration，避免歷史資料補值阻塞部署。
- [管理員查詢結果資料量成長] → 第一版先提供日期與員工篩選，預設以單日查詢為主要操作路徑，後續如有需要再補分頁。
- [代訂與員工自行訂餐規則漂移] → 建立流程重用既有訂單 service 的核心驗證，只在管理員入口額外附加角色與 `created_by` 行為。

## Migration Plan

- 先部署 `orders.created_by` nullable 欄位與外鍵 migration。
- 再部署 backend API 與 service，確保新欄位在代訂成功時可正確寫入。
- 最後部署 frontend 管理頁入口，避免前端先上線但後端尚未支援。

## Open Questions

- 查詢 API 是否需要第一版就加上分頁；若管理端預設僅查看單日資料，可先不納入 A006。
- 若管理員代訂遇到同員工同日既有訂單，第一版是否直接拒絕或改為提示改單流程；目前設計採直接拒絕並沿用 A002 既有規則。
