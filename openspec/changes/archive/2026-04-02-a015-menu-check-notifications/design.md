## Context

目前系統已經有 A004 錯誤通知信箱、A014 上班日行事曆，以及管理員供應商/菜單設定頁，但這些能力仍是分散的。A015 要求系統主動檢查「未來一個月的上班日是否已設定菜單」，並在每日排程與管理員登入兩條路徑下提供提醒，同時避免同一天重複發信或重複打擾。

依 `openspec/config.yaml` 的規則，後端 controller 應以需求編號命名，因此本需求的 API controller 應採 `A015Controller.java`。排程與資料存取則需沿用既有 `service` / `dao` 分層，並在 exception 路徑保留明確 log。需求來源以 `../uml/input/A015.md`、`../uml/output/A015/*` 與 `../uml/openapi.yaml` 為準。

## Goals / Non-Goals

**Goals:**
- 提供 A015 每日 8:00 檢查與 Email 提醒流程，且同日只發送一次。
- 提供管理員登入後即時檢查 API，回傳未設定菜單日期，並支援 dismiss 當日提醒。
- 讓前端在登入後可依檢查結果自動導向菜單設定頁，顯示彈窗與頂部警示橫幅。
- 明確定義 A015 與 A004、A014、既有 menu 資料之間的責任邊界。

**Non-Goals:**
- 不修改 `../uml` 內的需求文件、OpenAPI 或 UML。
- 不在本 change 內建立複雜的通知歷史查詢 UI。
- 不擴充簡訊、Slack 等其他提醒通道，僅處理既有 Email 與前端警示。
- 不改變菜單主檔本身的建立/修改 API 契約。

## Decisions

### 1. A015 使用兩張獨立資料表處理寄信與 dismiss
- 決策：依 UML 建立 `menu_notification_log` 與 `menu_notification_dismiss`。前者記錄每日寄信結果與缺漏日期範圍，後者記錄管理員是否已將當日提醒標記為已處理。
- 理由：A015 同時需要「同日只發一封信」與「當日登入後不再重複提醒」兩種狀態，分成兩張表最清楚，也符合 ER 圖。
- 替代方案：只用一張表加多種狀態欄位。未採用，因為寄信結果與前端 dismiss 語意不同，混在同一張表會增加查詢條件複雜度。

### 2. 缺漏日期以 A014 工作日為基礎，再比對菜單覆蓋範圍
- 決策：A015 先查 `work_calendar` 中未來一個月的 `is_workday=true` 日期，再比對這些日期是否存在至少一筆有效菜單覆蓋。
- 理由：需求明確指向「上班日是否有未設定菜單」，因此 A014 是這個需求的日期來源。
- 替代方案：直接依平日/週末推算，不查 `work_calendar`。未採用，因為會忽略人工調整過的上班日/非上班日。

### 3. 排程寄信與登入即時檢查共用同一套缺漏計算 service
- 決策：建立共用的 menu check service，排程與 `GET /api/admin/notifications/menu-check` 都透過同一個缺漏計算入口取得結果。
- 理由：這可避免「Email 寄出的缺漏日期」與「前端看到的缺漏日期」不一致。
- 替代方案：排程和 API 各自實作查詢。未採用，因為容易導致邏輯分岐與回歸風險。

### 4. dismiss 僅作用於當日登入提醒，不回頭修改寄信記錄
- 決策：`POST /api/admin/notifications/menu-check/dismiss` 僅記錄當日已處理，影響同日後續登入時是否再顯示提醒，不反寫既有寄信 log。
- 理由：需求寫的是「管理員補完菜單後當天不再提醒」，不是撤銷已寄出的排程通知。
- 替代方案：dismiss 時更新當日寄信 log 狀態。未採用，因為會讓寄信事實與 UI 處理狀態混淆。

### 5. 前端提醒掛在管理員登入後初始化流程，並導向既有菜單設定頁
- 決策：管理員登入成功後或刷新管理資料時，前端呼叫 `GET /api/admin/notifications/menu-check`。若存在未設定日期且當日未 dismiss，前端切換到既有菜單設定頁並顯示彈窗與橫幅。
- 理由：A015 指定「進入系統時即時檢查」與「跳轉至菜單設定頁」，最適合掛在管理員 console 初始化流程。
- 替代方案：只顯示角落提醒、不自動導頁。未採用，因為與需求不符。

## Risks / Trade-offs

- [若缺漏計算邏輯與菜單有效期間定義不一致，前端提醒與實際菜單頁會互相矛盾] → A015 應重用既有 menu 有效日規則，並以整合測試驗證代表性日期。
- [每日排程與登入即時檢查可能在同一天各自重算，造成額外查詢成本] → 共用 service，並將查詢視窗限制在未來一個月。
- [自動跳轉至菜單設定頁可能打斷管理員原本想看的頁面] → 透過明確 MessageBox 與頂部橫幅說明原因，並提供 dismiss 當日提醒的操作。
- [若錯誤通知信箱未設定，排程可能無法發信] → 需定義可觀測的失敗 log，並將結果寫入 `menu_notification_log`。

## Migration Plan

- 新增 `menu-check-notification` 主規格，定義缺漏檢查、每日寄信、登入即時檢查與 dismiss 契約。
- 更新 `error-email-settings`，明確說明 A004 收件清單同時供 A015 使用。
- 更新 `frontend-tab-navigation-and-message-box`，補上 A015 導頁、彈窗與頂部警示規則。
- 實作提醒資料表、repository、service、排程與 `A015Controller`。
- 實作前端管理員登入後的即時檢查與提醒 UI。
