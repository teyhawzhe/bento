## Context

目前前端主規格已定義管理員登入後會看到固定主 TAB，且 CSV 匯入已經被明確提升為與系統設定同層級的主導覽項目。不過實作上，員工帳號管理仍留在「系統設定」中的 `employee-create` 子頁，與錯誤通知信箱、部門管理共用同一個容器。這使得「員工管理」這個高頻操作在資訊架構上顯得太深，也讓系統設定區塊變成混合主檔維護與系統設定的集合。

這次 change 的重點不是改動員工 API，也不是重新設計整個後台，而是把既有員工帳號流程提升到正確的導覽層級，讓主導覽能更準確反映管理員的工作分區。

## Goals / Non-Goals

**Goals:**
- 在管理員主導覽新增獨立的「員工管理」主 TAB。
- 將新增員工、員工清單、部門篩選、編輯、啟停與重設密碼整合到「員工管理」主 TAB。
- 讓系統設定頁保留設定型功能，避免再承載員工帳號主流程。
- 維持現有員工管理 API、MessageBox 行為與單頁切換模式不變。

**Non-Goals:**
- 不新增或修改任何員工管理後端 API。
- 不在這次 change 中重新定義部門管理是否也要升級為主 TAB。
- 不重做整體視覺設計，只調整資訊架構與內容分區。
- 不變更管理員登入後的預設 landing tab；若目前預設為「訂單管理」，則保持不變。

## Decisions

### 1. 新增「員工管理」主 TAB，作為所有員工帳號操作的唯一入口
- 決策：在管理員主 TAB 列加入 `員工管理`，並把原本放在系統設定中的員工相關內容整體搬移到此主 TAB。
- 理由：新增員工、查詢員工、編輯與停用都是日常主檔維護，不屬於低頻的系統設定。放到主 TAB 更符合使用頻率與心智模型。
- 替代方案：保留在系統設定子頁，只調整文案。未採用，因為核心問題在導覽層級，而不是命名。

### 2. 「員工管理」TAB 內承載建立與查詢維護兩塊內容
- 決策：新主 TAB 直接承載目前已有的新增員工表單與員工清單區塊，包含部門篩選、編輯、啟停與重設密碼。
- 理由：這些操作共享同一組員工資料脈絡與部門資料來源，留在同一個主頁面最直觀，也能減少不必要的二級切換。
- 替代方案：再把「新增員工」與「員工清單」切成主 TAB 下的子 TAB。未採用，因為目前資料量與互動複雜度尚不足以需要再分層。

### 3. 「系統設定」保留錯誤通知信箱與部門管理，不再承載員工帳號工具
- 決策：移除 `employee-create` 這類員工帳號子頁，系統設定聚焦在錯誤通知信箱、部門管理與其他設定性質功能。
- 理由：部門管理雖與員工建立有依賴，但本質上仍較接近系統主檔設定；本次 user request 也只要求把員工相關功能移到主 TAB。
- 替代方案：把部門管理也一起搬到員工管理。暫不採用，避免在未被明確要求前擴大範圍。

### 4. 管理員主 TAB 數量擴充為六個，既有切換方式不變
- 決策：主導覽更新為「訂單管理」「供應商管理」「報表設定」「員工管理」「CSV 匯入」「系統設定」六個主 TAB，仍使用既有單頁 state 切換。
- 理由：這能最小成本延續 A009 之後建立的前端 shell，且不需要新增 router 或改寫登入導向。
- 替代方案：以「系統設定」主 TAB 內增加更醒目的次導覽。未採用，因為仍然沒有解決主層級不足的問題。

## Risks / Trade-offs

- [管理員主導覽再增加一個 TAB 後，桌面與窄畫面寬度會更緊] → 實作時應確認 TabBar 在較窄寬度下仍可換行或橫向捲動，避免字串被擠壓。
- [員工管理內容從系統設定搬出後，若 state 初始化與資料載入條件綁太深，可能造成切換後空白或沒 reload] → 保持既有資料載入邏輯，以主 TAB 切換只影響顯示區塊，不改動 API 觸發責任。
- [部門管理仍留在系統設定，使用者可能預期也一起搬動] → 在規格中明確寫出本次只提升員工帳號功能，不調整部門管理層級。

## Migration Plan

- 先更新 OpenSpec，明確定義管理員主導覽新增「員工管理」TAB。
- 在前端將原本系統設定中的員工帳號工具區塊搬到新的主 TAB，保留既有表單與清單行為。
- 收斂系統設定頁內容，移除員工帳號子頁定義與相關切換狀態。
- 驗證管理員登入後可進入新主 TAB，且新增、查詢、篩選、編輯、啟停與重設密碼流程都維持正常。

## Open Questions
